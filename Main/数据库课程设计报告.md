# 第一部分、开发前的设计和思考

## 一、项目背景

### 1.1 项目概述

QuillCode 是一个面向程序员的在线代码笔记系统，旨在提供一个"随写、随存、随运行"的代码学习与管理环境。系统支持 Markdown 文档编写、多语言代码编辑与在线执行、标签分类管理、智能文章关联与推荐、文章分享与评论等功能。

"Quill" 一词源自英文中的"羽毛笔"，象征着书写与创作。在 QuillCode 中，每一篇代码笔记都被称为一个 "Quill"，寓意用户用代码书写自己的技术成长之路。

### 1.2 开发背景

随着编程学习的普及和技术迭代的加速，程序员面临着知识管理的巨大挑战：

1. **碎片化学习困境**：学习资料分散在各种平台，难以系统整理
2. **笔记与代码分离**：传统笔记软件无法执行代码，而 IDE 又不适合做笔记管理
3. **知识复用困难**：曾经写过的代码片段难以快速检索和复用
4. **缺乏智能关联**：相关知识点之间缺乏有效的关联和推荐机制

QuillCode 应运而生，致力于解决上述痛点，将 Markdown 文档与可执行代码完美结合，打造程序员的"第二大脑"。

### 1.3 项目目标

本项目的核心目标是构建一个功能完善、体验流畅的代码笔记管理平台：

| 目标类别 | 具体目标 | 实现方式 |
|---------|---------|---------|
| 内容管理 | 提供 Markdown + 代码的统一管理平台 | 富文本编辑器 + 代码编辑器集成 |
| 代码执行 | 支持 JavaScript、Python、Java 等多语言代码在线执行 | 后端沙箱执行环境 |
| 智能搜索 | 实现基于 Elasticsearch 的全文模糊搜索 | ES 索引 + 多字段匹配 |
| 智能推荐 | 提供基于标签和 AI 的智能推荐功能 | 标签相似度 + Ollama 大模型 |
| 社交分享 | 支持文章分享与访客评论功能 | Token 分享链接 + 评论系统 |
| 系统管理 | 提供管理员统计与维护功能 | 数据统计 + 日志审计 |

## 二、需求分析

### 2.1 功能需求

#### 2.1.1 用户管理模块

用户管理是系统的基础模块，负责用户身份的认证与授权。

```mermaid
flowchart LR
    subgraph 用户管理模块
        A[用户注册] --> B[信息验证]
        B --> C[密码加密]
        C --> D[创建账户]
        E[用户登录] --> F[身份验证]
        F --> G[生成JWT]
        G --> H[返回Token]
    end
```

| 功能点 | 描述 | 技术实现 |
|-------|------|---------|
| 用户注册 | 支持用户名、邮箱、密码注册 | bcrypt 密码加密 |
| 用户登录 | JWT Token 认证机制 | 7天有效期 Token |
| 角色区分 | 普通用户与管理员 | ENUM 字段 + 守卫验证 |
| 信息校验 | 用户名/邮箱唯一性检查 | 数据库唯一约束 |

#### 2.1.2 文章管理模块

文章（Quill）是系统的核心数据实体，支持多种类型的代码笔记管理。

```mermaid

        F --> I[级联删除]

```

| 功能点 | 描述 | 关联操作 |
|-------|------|---------|
| 创建文章 | 支持标题、Markdown 内容、代码、类型、语言、标签 | 同步 ES 索引 |
| 文章类型 | 算法题解(algorithm)、代码片段(snippet)、HTML页面(html) | 前端差异化展示 |
| 编辑更新 | 支持实时保存和更新 | 触发器记录日志 |
| 删除文章 | 级联删除相关分享和标签关联 | 外键 CASCADE |

#### 2.1.3 标签管理模块

标签系统采用用户隔离模式，确保用户数据的私密性。

```mermaid

```

| 功能点 | 描述 | 设计要点 |
|-------|------|---------|
| 标签创建 | 支持自定义标签（用户隔离模式） | user_id + name 复合唯一 |
| 用户隔离 | 每个用户拥有独立的标签空间，互不可见 | 查询时过滤 user_id |
| 标签关联 | 多对多关系，一篇文章可有多个标签 | article_tags 中间表 |
| 标签筛选 | 按标签查看文章列表 | JOIN 查询 |
| 权限控制 | 用户只能管理自己的标签 | 服务层权限校验 |

#### 2.1.4 搜索模块

基于 Elasticsearch 实现高性能全文搜索。


|-------|------|---------|

| 搜索范围 | 标题、内容、代码、标签 | 多字段权重配置 |
| 高亮显示 | 搜索结果关键词高亮 | highlight 配置 |
| 降级策略 | ES 不可用时降级到 MySQL | try-catch 容错 |

智能推荐系统结合标签相似度和 AI 大模型。

## 三、概念结构设计

### 3.1 实体设计

通过需求分析，采用自顶向下的方法设计出系统的核心实体。实体设计遵循以下原则：

1. **独立性原则**：每个实体应具有独立存在的意义
2. **完整性原则**：实体应包含描述其特征的完整属性
3. **最小冗余原则**：避免实体间的属性重复

```mermaid
mindmap
  root((QuillCode 实体))
    核心实体
      User 用户
      Article 文章
      Tag 标签
    关联实体
      ArticleTag 文章标签
    扩展实体
      Share 分享
      Comment 评论
    系统实体
      OperationLog 操作日志
```

通过需求分析，设计出以下核心实体：

| 实体名称 | 说明 | 主要属性 | 实体类型 |
|---------|------|----------|---------|
| User (用户) | 系统用户，包含普通用户和管理员 | id, username, email, password_hash, role | 核心实体 |
| Article (文章) | 代码笔记文章，系统核心数据 | id, title, content, code, type, language | 核心实体 |
| Tag (标签) | 文章分类标签，用户隔离模式 | id, user_id, name | 核心实体 |
| Share (分享) | 文章分享链接，支持过期时间 | id, token, expires_at | 扩展实体 |
| Comment (评论) | 分享页面的访客评论 | id, author_name, content | 扩展实体 |
| OperationLog (操作日志) | 系统操作记录，审计追踪 | id, table_name, operation_type, record_id | 系统实体 |

### 4.3 关系模式规范化分析

#### 4.3.1 规范化流程

```mermaid
flowchart LR
    A[原始关系] --> B{满足1NF?}
    B -->|是| C{满足2NF?}
    B -->|否| B1[消除非原子属性]
    B1 --> B
    C -->|是| D{满足3NF?}
    C -->|否| C1[消除部分依赖]
    C1 --> C
    D -->|是| E{满足BCNF?}
    D -->|否| D1[消除传递依赖]
    D1 --> D
    E -->|是| F[规范化完成]
    E -->|否| E1[消除主属性依赖]
    E1 --> E
```

#### 4.3.2 第一范式 (1NF) 验证

**定义**：关系模式R的所有属性都是不可再分的基本数据项。

| 关系模式 | 验证结果 | 说明 |
|---------|---------|------|
| users | ✅ 满足1NF | 所有字段都是原子值 |
| articles | ✅ 满足1NF | content和code虽为TEXT，但作为整体存储 |
| tags | ✅ 满足1NF | name是单值属性 |
| article_tags | ✅ 满足1NF | 联合主键，无其他属性 |
| shares | ✅ 满足1NF | 所有字段都是原子值 |
| comments | ✅ 满足1NF | 所有字段都是原子值 |
| operation_logs | ✅ 满足1NF | JSON字段作为整体存储 |

#### 4.3.3 第二范式 (2NF) 验证

**定义**：在满足1NF的基础上，消除非主属性对候选键的部分函数依赖。

| 关系模式 | 验证结果 | 说明 |
|---------|---------|------|
| users | ✅ 满足2NF | 单一主键，不存在部分依赖 |
| articles | ✅ 满足2NF | 单一主键，不存在部分依赖 |
| tags | ✅ 满足2NF | 单一主键，不存在部分依赖 |
| article_tags | ✅ 满足2NF | 联合主键(article_id, tag_id)，无非主属性 |
| shares | ✅ 满足2NF | 单一主键，不存在部分依赖 |
| comments | ✅ 满足2NF | 单一主键，不存在部分依赖 |
| operation_logs | ✅ 满足2NF | 单一主键，不存在部分依赖 |

#### 4.3.4 第三范式 (3NF) 验证

**定义**：在满足2NF的基础上，消除非主属性对候选键的传递函数依赖。

```mermaid
flowchart TB
    subgraph 3NF验证-传递依赖检查

        A1 -->|通过外键关联| A2[✅ 传递依赖消除]

```

| 关系模式 | 验证结果 | 说明 |
|---------|---------|------|
| users | ✅ 满足3NF | 所有非主属性直接依赖于主键 |
| articles | ✅ 满足3NF | user_id → username 的传递依赖通过外键关联消除 |
| tags | ✅ 满足3NF | 所有非主属性直接依赖于主键 |
| article_tags | ✅ 满足3NF | 无非主属性，自然满足 |
| shares | ✅ 满足3NF | article_id → title 的传递依赖通过外键关联消除 |
| comments | ✅ 满足3NF | share_id → article_id 的传递依赖通过外键关联消除 |
| operation_logs | ✅ 满足3NF | 所有非主属性直接依赖于主键 |

#### 4.3.5 BCNF 验证

**定义**：在满足3NF的基础上，对于每个非平凡函数依赖X→Y，X都是超键。

所有关系模式均满足BCNF，因为：
- 每个函数依赖的决定因素都是候选键或超键
- 不存在主属性对非主属性的依赖

### 5.4 存储过程设计

#### 5.4.1 获取用户文章列表 (sp_get_user_articles)

**用途**：分页获取指定用户的文章列表，支持按类型筛选。

```sql
DELIMITER //
CREATE PROCEDURE `sp_get_user_articles`(
  IN p_user_id CHAR(36),
  IN p_page INT,
  IN p_page_size INT,
  IN p_type VARCHAR(20)
)
BEGIN
  DECLARE v_offset INT;
  SET v_offset = (p_page - 1) * p_page_size;
  
  SELECT 
    a.id,
    a.title,
    a.type,
    a.language,
    a.created_at,
    a.updated_at,
    (SELECT GROUP_CONCAT(t.name) FROM article_tags at 
     JOIN tags t ON at.tag_id = t.id 
     WHERE at.article_id = a.id) AS tags
  FROM articles a
  WHERE a.user_id = p_user_id
    AND (p_type IS NULL OR p_type = '' OR a.type = p_type)
  ORDER BY a.updated_at DESC
  LIMIT v_offset, p_page_size;
END //
DELIMITER ;
```

#### 5.4.2 创建文章并关联标签 (sp_create_article_with_tags)

**用途**：事务性地创建文章并关联多个标签，保证数据一致性。

```sql
DELIMITER //
CREATE PROCEDURE `sp_create_article_with_tags`(
  IN p_article_id CHAR(36),
  IN p_user_id CHAR(36),
  IN p_title VARCHAR(255),
  IN p_content TEXT,
  IN p_code TEXT,
  IN p_type VARCHAR(20),
  IN p_language VARCHAR(50),
  IN p_tag_ids TEXT  -- 逗号分隔的标签ID
)
BEGIN
  DECLARE v_tag_id CHAR(36);
  DECLARE v_pos INT;
  DECLARE v_remaining TEXT;
  
  DECLARE EXIT HANDLER FOR SQLEXCEPTION
  BEGIN
    ROLLBACK;
    RESIGNAL;
  END;
  
  START TRANSACTION;
  
  -- 插入文章
  INSERT INTO articles (id, user_id, title, content, code, type, language)
  VALUES (p_article_id, p_user_id, p_title, p_content, p_code, p_type, p_language);
  
  -- 解析并插入标签关联
  IF p_tag_ids IS NOT NULL AND p_tag_ids != '' THEN
    SET v_remaining = p_tag_ids;
    
    WHILE LENGTH(v_remaining) > 0 DO
      SET v_pos = LOCATE(',', v_remaining);
      
      IF v_pos > 0 THEN
        SET v_tag_id = TRIM(SUBSTRING(v_remaining, 1, v_pos - 1));
        SET v_remaining = SUBSTRING(v_remaining, v_pos + 1);
      ELSE
        SET v_tag_id = TRIM(v_remaining);
        SET v_remaining = '';
      END IF;
      
      IF v_tag_id != '' THEN
        INSERT IGNORE INTO article_tags (article_id, tag_id) VALUES (p_article_id, v_tag_id);
      END IF;
    END WHILE;
  END IF;
  
  COMMIT;
  
  SELECT 'SUCCESS' AS result, p_article_id AS article_id;
END //
DELIMITER ;
```

#### 5.4.3 获取用户统计数据 (sp_get_user_stats)

**用途**：获取指定用户的综合统计数据。

```sql
DELIMITER //
CREATE PROCEDURE `sp_get_user_stats`(
  IN p_user_id CHAR(36)
)
BEGIN
  SELECT 
    u.username,
    COUNT(DISTINCT a.id) AS total_articles,
    COUNT(DISTINCT s.id) AS total_shares,
    COUNT(DISTINCT c.id) AS total_comments,
    (SELECT COUNT(DISTINCT at.tag_id) FROM articles a2 
     JOIN article_tags at ON a2.id = at.article_id 
     WHERE a2.user_id = p_user_id) AS tags_used
  FROM users u
  LEFT JOIN articles a ON u.id = a.user_id
  LEFT JOIN shares s ON a.id = s.article_id
  LEFT JOIN comments c ON s.id = c.share_id
  WHERE u.id = p_user_id
  GROUP BY u.id, u.username;
END //
DELIMITER ;
```

#### 5.4.4 清理过期分享 (sp_cleanup_expired_shares)

**用途**：定期清理过期的分享链接，评论会级联删除。

```sql
DELIMITER //
CREATE PROCEDURE `sp_cleanup_expired_shares`()
BEGIN
  DECLARE v_deleted_count INT DEFAULT 0;
  
  -- 删除过期的分享（评论会级联删除）
  DELETE FROM shares WHERE expires_at < NOW();
  
  SET v_deleted_count = ROW_COUNT();
  
  SELECT v_deleted_count AS deleted_shares, NOW() AS cleanup_time;
END //
DELIMITER ;
```

```typescript
// backend/src/modules/search/search.service.ts
async search(query: string, userId?: string): Promise<SearchResult[]> {
  if (!this.isConnected) {
    return [];
  }

  const must: object[] = [
    {
      multi_match: {
        query,
        fields: ['title^3', 'content^2', 'code', 'tags^2'],
        type: 'best_fields',
        fuzziness: 'AUTO',  // 自动模糊匹配
      },
    },
  ];

  // 限制只搜索当前用户的文章
  if (userId) {
    must.push({ term: { userId } });
  }

  const response = await this.client.search<ArticleDocument>({
    index: this.indexName,
    query: {
      bool: { must },
    },
    highlight: {
      fields: {
        title: { number_of_fragments: 1 },
        content: { number_of_fragments: 2, fragment_size: 150 },
        code: { number_of_fragments: 1, fragment_size: 200 },
      },
      pre_tags: ['<mark>'],
      post_tags: ['</mark>'],
    },
    size: 20,
  });

  return response.hits.hits.map((hit) => ({
    id: hit._source?.id || '',
    title: hit._source?.title || '',
    content: hit._source?.content || '',
    type: hit._source?.type || '',
    language: hit._source?.language || '',
    tags: hit._source?.tags || [],
    highlights: {
      title: hit.highlight?.title,
      content: hit.highlight?.content,
      code: hit.highlight?.code,
    },
    score: hit._score || 0,
  }));
}
```

```typescript
// 创建文章并关联标签（使用事务）
async create(userId: string, createArticleDto: CreateArticleDto): Promise<Article> {
  const { tagIds, ...articleData } = createArticleDto;

  // 开始事务
  const queryRunner = this.dataSource.createQueryRunner();
  await queryRunner.connect();
  await queryRunner.startTransaction();

  try {
    // 1. 创建文章
    const article = this.articleRepository.create({
      ...articleData,
      userId,
    });
    const savedArticle = await queryRunner.manager.save(article);

    // 2. 关联标签
    if (tagIds && tagIds.length > 0) {
      const articleTags = tagIds.map(tagId => ({
        articleId: savedArticle.id,
        tagId,
      }));
      await queryRunner.manager.insert(ArticleTag, articleTags);
    }

    // 3. 提交事务
    await queryRunner.commitTransaction();

    // 4. 索引到 Elasticsearch
    await this.searchService.indexArticle(savedArticle);

    return savedArticle;
  } catch (error) {
    // 回滚事务
    await queryRunner.rollbackTransaction();
    throw error;
  } finally {
    await queryRunner.release();
  }
}
```

**核心代码实现**：

```typescript
async getRecommendations(articleId: string, userId: string) {
  // 1. 获取当前文章的标签
  const article = await this.articleRepository.findOne({
    where: { id: articleId },
    relations: ['articleTags', 'articleTags.tag'],
  });
  
  const tagIds = article.articleTags.map(at => at.tagId);
  
  // 2. 查找具有相同标签的其他文章
  const recommendations = await this.articleRepository
    .createQueryBuilder('article')
    .leftJoinAndSelect('article.articleTags', 'at')
    .leftJoinAndSelect('at.tag', 'tag')
    .where('article.userId = :userId', { userId })
    .andWhere('article.id != :articleId', { articleId })
    .andWhere('at.tagId IN (:...tagIds)', { tagIds })
    .groupBy('article.id')
    .orderBy('COUNT(at.tagId)', 'DESC')  // 按共同标签数量排序
    .limit(5)
    .getMany();
  
  return recommendations;
}
```

**问题描述**：需要记录所有增删改操作，但不想在每个业务方法中手动添加。

**解决方案**：使用数据库触发器自动记录：

```sql
CREATE TRIGGER `tr_article_after_insert`
AFTER INSERT ON `articles`
FOR EACH ROW
BEGIN
  INSERT INTO operation_logs (
    id, table_name, operation_type, record_id, new_data, operated_at
  ) VALUES (
    UUID(), 'articles', 'INSERT', NEW.id, 
    JSON_OBJECT('title', NEW.title, 'type', NEW.type),
    NOW()
  );
END;
```

### 2.6 问题五：标签数据隔离

```mermaid
flowchart TB
    subgraph 问题-全局共享模式
        A1[用户A创建标签] --> B1[全局可见]
        A2[用户B] --> B1
        B1 --> C1[隐私泄露风险]
    end
    
    subgraph 解决方案-用户隔离模式
        D1[用户A创建标签] --> E1[仅A可见]
        D2[用户B创建标签] --> E2[仅B可见]
        E1 --> F1[数据隔离]
        E2 --> F1
    end
```

**问题描述**：原有设计中标签是全局共享的，导致用户能看到其他用户的标签及其文章数量统计，存在隐私泄露风险。

**解决方案**：将标签改为用户隔离模式：

```mermaid
flowchart LR
    subgraph 数据库变更
        A[添加user_id字段] --> B[修改唯一约束]
        B --> C[添加外键约束]
        C --> D[更新视图]
    end
```

1. 在 tags 表中添加 `user_id` 外键字段
2. 将原有的 `uk_name` 唯一约束改为 `uk_user_tag(user_id, name)` 复合唯一约束
3. 修改 TagService 的所有方法，增加 userId 参数进行过滤
4. 更新 v_popular_tags 视图，增加 user_id 字段支持用户级别统计

```sql
-- 标签表结构变更
ALTER TABLE `tags` ADD COLUMN `user_id` CHAR(36) NOT NULL;
ALTER TABLE `tags` DROP INDEX `uk_name`;
ALTER TABLE `tags` ADD UNIQUE KEY `uk_user_tag` (`user_id`, `name`);
ALTER TABLE `tags` ADD CONSTRAINT `fk_tags_user` 
  FOREIGN KEY (`user_id`) REFERENCES `users` (`id`) ON DELETE CASCADE;
```

**隔离效果对比**：

| 对比项 | 全局共享模式 | 用户隔离模式 |
|-------|------------|------------|
| 标签可见性 | 所有用户可见 | 仅创建者可见 |
| 同名标签 | 不允许 | 不同用户可创建同名 |
| 隐私保护 | ❌ 无 | ✅ 有 |
| 唯一约束 | name | (user_id, name) |
